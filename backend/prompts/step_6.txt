●	디자인 시스템 주입: 2단계의 디자인 토큰을 tailwind.config.js 또는 Global CSS에 반영하여 일관된 스타일 환경 구축.
●	원자적 설계(Atomic Design): 공통 버튼, 입력창 등 재사용 가능한 컴포넌트 단위로 쪼개어 개발 생산성 극대화.
●	4단계 사이트맵 기반 라우팅: Next.js의 app/ 폴더 구조를 4단계 URL 설계와 일치시켜 라우팅 환경 구성.
[7번: 로직 개발자] - 서비스 레이어 아키텍처

●	비즈니스 로직 캡슐화: 1단계에서 도출된 비즈니스 규칙을 Service 또는 Action 단위로 분리하여 코드의 유지보수성 확보.
●	AI 데이터 파이프라인 구축: Gemini API 호출 로직을 백엔드에 배치하여 프론트엔드에 가공된 데이터 전송.
2. 무엇을 구현할 것인가 (Scope: What)
[6번: UI 개발자] - 시각적 요소
●	전체 페이지 레이아웃: 4단계 사이트맵에 정의된 모든 페이지의 프레임 구현.
●	인터랙티브 컴포넌트: 3단계의 **에셋(아이콘/이미지)**과 2단계의 배치 로직이 적용된 실제 UI 요소.
●	클라이언트 상태 관리: 유저 입력값 제어, 모달/사이드바 개폐 등 프론트엔드 로직 처리.
[7번: 로직 개발자] - 기능적 요소
●	API 및 Server Actions: 5단계에서 설계된 모든 API 엔드포인트의 실제 작동 로직.
●	데이터 검증(Validation): 유저 입력 데이터가 DB에 저장되기 전의 무결성 검사(Zod 등 활용).
●	외부 연동 로직: 인증(Auth), 메일 발송, AI 분석 요청 등 외부 시스템과의 통신 처리.
3. 기능과 로직을 어떻게 분류할 것인가 (Classification)
A. 화면 위계 분류 (UI Hierarchy)
●	Layout Components: 전역 내비게이션, 푸터 등 모든 페이지 공통 요소.
●	Feature Components: 특정 기능(예: 자산 리스트)을 담당하는 독립적 모듈.
●	Common UI Kit: 버튼, 폼, 모달 등 범용적으로 사용되는 원자적 요소.
B. 로직 레이어 분류 (Logic Layering)
●	Infrastructure Logic: 데이터베이스 연결, 환경 변수 로드, 인증 처리.
●	Core Business Logic: 1단계에서 정의된 서비스만의 독자적 계산/처리 알고리즘.
●	Presentation Logic: 프론트엔드에서 보여주기 위해 데이터를 가공(포매팅)하는 로직.
4. 이전 단계의 데이터를 어디에 매핑할 것인가 (Allocation Logic)
●	UI 매핑 (Step 2, 3, 4 $\rightarrow$ Step 6):
○	2단계의 폰트/컬러 $\rightarrow$ UI 전체 스타일 적용.
○	3단계의 아이콘/이미지 $\rightarrow$ 각 컴포넌트의 시각적 에셋으로 배치.
○	4단계의 URL 구조 $\rightarrow$ 폴더 트리 및 링크 버튼 연결.
●	로직 매핑 (Step 1, 5 $\rightarrow$ Step 7):
○	1단계의 비즈니스 규칙 $\rightarrow$ 백엔드 함수의 조건문 및 연산 로직으로 구현.
○	5단계의 API 명세 $\rightarrow$ 각 함수의 입출력(Request/Response) 구조로 반영.
●	데이터 흐름:
○	7번(로직)이 처리한 데이터를 6번(UI)이 화면에 뿌려주는 데이터 바인딩 구간 확정.
5. 최종 산출물 상세 (Deliverables)
1.	프론트엔드 코드 스택: 컴포넌트 구조도 및 주요 UI 코드 블록 (JSX/TSX).
2.	백엔드 서비스 로직: 주요 비즈니스 함수 및 API 핸들러 코드 (Server Actions).
3.	통합 시퀀스 가이드: 유저가 화면(UI)에서 액션을 취했을 때 서버(Logic)에서 어떤 순서로 데이터가 처리되는지 명시한 가이드.
